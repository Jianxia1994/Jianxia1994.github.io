open-falcon初识

2018-12-17
Development golang index open-falcon

Table of Contents

    • 组件

优点

  1. 自动发现比较好用，支持用户主动push，时间序列数据存储
  2. 水平扩展能力好
  3. 告警配置较为方便，支持callback调用
  4. 主机、metric等的层级更灵活
  5. 多维度数据展示、用户可以自定义screen
  6. 高可用比zabbix好很多，整个系统没有核心单点，可水平扩展
  7. 后端全部golang，portal和dashboard 为python，适合运维进行二次开发（对比zabbix的php和C）
  8. 针对各个开源软件都有自己的监控插件，比zabbix全面的多，如果没有响应的插件，自己开发也比较容易。插件是独立于agent进程的，安装新插件不需要重启agent
总之，open-falcon不是一个监控软件，而是一个运维框架，只要把数据整合成标准的形式就可以接收和配置告警，收数据有点类似于prometheus，但是他的告警又远远优于prometheus

组件

  1. agent
    监控客户端，相当于zabbix_agent，默认自带了很多监控项不需要配置，同时还提供接口供自定义脚本上报数据 支持自发现，自定义监控指标通过HBS获取 每隔60s通过agent与transfer建立的长连接吧数据push给transfer，数据发送速度快
  2. Aggregater 集群聚合模块
    聚合集群下所有机器的某个指标的值，比如所有机器的qps才是整个集群的qps，平均的某个值才是集群的某个值，计算完成后push回server端。
  3. hbs（heartbeat server）
    agent每分钟发一次心跳信息给hbs，同时会带上hostname、ip、agent version、plugin version等信息
  4. portal
    配置报警策略，hbs读取数据库中的策略传给agent
  5. transfer
    数据转发服务，收到agent上报的数据后，按照哈希规则分片，将分片后的数据分别push给judge和graph
  6. judge
    告警判断，agent数据传给transfer，transfer给graph绘图，同时给judge判断是否触发告警
  7. alarm
    judge产生的报警event写入redis，alarm从redis中读取处理
  8. graph
    存储绘图数据。接收transfer推送的监控数据，也处理query的查询请求，返回绘图数据
  9. query
    提供统一的绘图数据查询入口。query组件接收查询请求，根据一致性哈希去响应的graph实例查询不同metric的数据，汇总数据返回给用户
  10. dashbord
    查询和配置界面
  11. nodata
    检测数据异常，与judge协同工作。配置了nodata的metric在数据缺失的情况下，会自动生成默认模拟数据，供judge检测，使告警更健壮
  12. task
    定时任务。index更新，包括图表索引的全量更新和垃圾索引清理。falcon服务组件的自身状态收集，falcon自监控任务。
